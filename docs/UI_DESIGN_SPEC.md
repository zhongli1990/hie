## 1. Executive Summary

This document defines the enterprise-grade UI design for the HIE Management Portal. The design connects all UI tabs to real backend APIs, provides real-time visibility into message flows, and supports comprehensive configuration management.

### Design Principles

| Principle | Description |
|-----------|-------------|
| **Real Data** | All UI components connected to live backend APIs |
| **Permanent Storage** | Messages stored in PostgreSQL permanently (with manual/automated housekeeping) |
| **Polling First** | Start with refresh button + polling, WebSocket in future phase |
| **Simple Charts** | Basic chart components with scalable placeholders for future enhancement |
| **Sub-Tab Navigation** | Configure tab uses sub-tabs for different configuration domains |

### Key Decisions

1. **Message Storage:** Permanent in PostgreSQL database with manual UI or automated housekeeping tasks for purging
2. **Configure Tab:** Sub-tab menu structure for Projects, Schemas, Transformations (DTL), Routing Rules, etc.
3. **Real-time Updates:** Polling with refresh button (simpler to start), WebSocket in later phase
4. **Charts:** Simple inline charts with scalable placeholders for advanced charting libraries

### Components

| Component | Description | Data Source |
|-----------|-------------|-------------|
| **Search Bar** | Full-text search across message content | Query parameter |
| **Filter Dropdowns** | Project, Item, Status, Time Range | Query parameters |
| **Message Table** | Paginated list with key columns | `GET /api/messages` |
| **Message Detail** | Slide-over with full content | `GET /api/messages/{id}` |
| **HL7 Viewer** | Color-coded segment display | Parsed from raw content |
| **Parsed View** | JSON tree view of HL7 fields | Parsed from raw content |

### HL7 Segment Color Coding

| Segment | Color | Description |
|---------|-------|-------------|
| MSH | Blue | Message Header |
| EVN | Cyan | Event Type |
| PID | Green | Patient Identification |
| PV1 | Yellow | Patient Visit |
| OBR | Orange | Observation Request |
| OBX | Pink | Observation Result |
| MSA | Purple | Message Acknowledgment |
| ERR | Red | Error Segment |

### Interactions

| Action | Behavior |
|--------|----------|
| Click message row | Open detail slide-over |
| Click "Resend" | Re-queue message for processing |
| Click "Trace" | Show message flow through all items |
| Click item metrics | Navigate here with item filter applied |
| Refresh button | Reload message list |
| Auto-refresh | Poll every 5 seconds when enabled |

### 7.1 Database Schema for Messages

```sql
-- Message storage table
CREATE TABLE messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID NOT NULL REFERENCES projects(id),
    item_name VARCHAR(255) NOT NULL,
    message_type VARCHAR(50),
    correlation_id VARCHAR(255),
    status VARCHAR(20) NOT NULL DEFAULT 'received',
    raw_content BYTEA NOT NULL,
    parsed_content JSONB,
    source_item VARCHAR(255),
    destination_item VARCHAR(255),
    size_bytes INTEGER,
    received_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    completed_at TIMESTAMP WITH TIME ZONE,
    latency_ms INTEGER,
    error_message TEXT,
    ack_content BYTEA,
    metadata JSONB DEFAULT '{}',
    
    -- Indexes for common queries
    INDEX idx_messages_project (project_id),
    INDEX idx_messages_item (item_name),
    INDEX idx_messages_status (status),
    INDEX idx_messages_type (message_type),
    INDEX idx_messages_received (received_at),
    INDEX idx_messages_correlation (correlation_id)
);
```

### 7.3 Message Storage Integration

Messages should be stored when:
1. **Inbound:** When a service receives a message
2. **Outbound:** When an operation sends a message
3. **ACK:** When an ACK is received (update existing record)
4. **Error:** When processing fails (update with error)
